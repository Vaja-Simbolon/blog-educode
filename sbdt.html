Materi Sistem Basis Data - Terdistribusi

Apa itu Sistem Basis Data - Terdistribusi

Sistem Basis Data adalah kumpulan data yang terorganisir serta perangkat lunak (software) yang dirancang untuk mengelola data tersebut. Sistem ini memungkinkan pengguna untuk menyimpan, mengelola, mengambil, dan memanipulasi data secara efisien. Sistem basis data digunakan untuk mendukung berbagai kebutuhan aplikasi, baik skala kecil hingga besar. Sedangkan Sistem Basis Data Terdistribusi (Distributed Database System) adalah sistem basis data di mana data disimpan di beberapa lokasi yang berbeda, namun dikelola sedemikian rupa sehingga terlihat seperti satu kesatuan. Data di setiap lokasi dapat diakses melalui jaringan, dan sistem ini dirancang untuk memastikan ketersediaan, keandalan, dan konsistensi data di seluruh lokasi.

Ciri-ciri Sistem Basis Data Terdistribusi
  • Distribusi Data: Data disimpan di lebih dari satu lokasi fisik.
  • Transparansi: Pengguna tidak perlu mengetahui di mana data sebenarnya disimpan.
  • Replikasi: Data dapat digandakan (di-replikasi) untuk meningkatkan ketersediaan dan kinerja.
  • Konektivitas Jaringan: Sistem membutuhkan jaringan untuk menghubungkan lokasi-lokasi penyimpanan data.
  • Independensi Lokasi: Pengguna dapat mengakses data tanpa memikirkan lokasi fisiknya.

Sistem ini sangat berguna untuk aplikasi berskala besar, seperti e-commerce, sistem perbankan, atau aplikasi yang melayani pengguna dari berbagai wilayah. xReplikasi dalam sistem basis data adalah proses membuat salinan data dari satu database ke database lain, biasanya untuk meningkatkan ketersediaan, keandalan, dan kinerja sistem. Dalam konteks Sistem Basis Data Terdistribusi (Distributed Database System), replikasi memainkan peran penting. Berikut penjelasan detailnya:

Apa itu Replikasi?

Replikasi adalah mekanisme di mana data dari satu server (master) disalin atau didistribusikan ke server lain (replica/slave). Proses ini bisa dilakukan secara otomatis berdasarkan konfigurasi tertentu, sehingga perubahan data pada server utama dapat direfleksikan pada salinan di server lain.

Tujuan Replikasi
  • Ketersediaan Data (Availability)
    Dengan replikasi, jika salah satu server gagal (down), data tetap bisa diakses dari salinan di server lain.
  • Kinerja yang Lebih Baik (Performance)
    Salinan data memungkinkan pembagian beban kerja (load balancing), sehingga banyak pengguna dapat mengakses data tanpa memengaruhi kinerja server utama.
  • Keamanan Data (Data Safety)
    Replikasi juga bisa digunakan sebagai mekanisme cadangan (backup) untuk melindungi data dari kerusakan atau kehilangan.
  • Distribusi Data
    Memungkinkan data tersedia lebih dekat dengan lokasi pengguna, mengurangi latensi jaringan.

Replikasi ada dua arah yaitu Master to Master dan Master to Slave

  • Replikasi Master to Master

    • Replikasi master-to-master adalah metode di mana dua atau lebih server database bertindak sebagai master, dan masing-masing dapat menerima pembaruan data. Setiap perubahan yang terjadi pada satu master akan direplikasi ke master lainnya.

    Karakteristik
    • Dua arah: Data dapat diubah di kedua server, dan perubahan akan disinkronkan di antara keduanya.
    • Konsistensi: Sangat cocok untuk sistem yang membutuhkan ketersediaan tinggi dan toleransi kegagalan.
    • Penggunaan: Biasanya digunakan untuk aplikasi yang membutuhkan kemampuan penulisan/pembacaan dari beberapa lokasi, seperti e-commerce atau aplikasi multi-lokasi.
    Kelebihan
    • Tidak ada single point of failure karena jika salah satu server gagal, server lainnya tetap bisa melayani.
    • Konsistensi: Sangat cocok untuk sistem yang membutuhkan ketersediaan tinggi dan toleransi kegagalan.
    Kekurangan
    • Konflik data dapat terjadi jika dua master melakukan perubahan pada data yang sama secara bersamaan. Biasanya diperlukan mekanisme untuk menyelesaikan konflik ini.
    • Konfigurasi dan pengelolaan lebih kompleks dibandingkan replikasi satu arah.

  • Replikasi Master to Slave

    • Replikasi master-to-slave adalah metode di mana hanya satu server yang bertindak sebagai master, sementara server lainnya bertindak sebagai slave. Master adalah satu-satunya sumber perubahan data, dan semua slave hanya menerima salinan (read-only) dari master.

    Karakteristik
    • Satu arah: Data hanya mengalir dari master ke slave.
    • Konsistensi baca: Slave digunakan untuk operasi pembacaan, sementara master menangani pembaruan data.
    • Penggunaan: Cocok untuk aplikasi yang membutuhkan banyak pembacaan data, seperti analitik atau cache data.
    Kelebihan
    • Beban kerja pembacaan dapat didistribusikan ke slave, sehingga master dapat lebih fokus pada operasi penulisan.
    • Relatif lebih sederhana untuk diatur dibandingkan master-to-master.
    Kekurangan
    • Tidak mendukung pembaruan data pada slave. Jika ada kebutuhan untuk pembaruan, perubahan harus dikirim kembali ke master.
    • Jika master gagal, sistem memerlukan konfigurasi ulang untuk menjadikan salah satu slave sebagai master (failover).

Perbedaan Utama
Fitur Master-to-Master Master-to-Slave
Arah Replikasi Dua arah Satu arah (Master ke Slave)
Operasi Tulis Semua server dapat menulis Hanya master yang dapat menulis
Kompleksitas Lebih kompleks (butuh manajemen konflik) Relatif lebih sederhana
Kegunaan Utama High Availability (multi-location writes) Load balancing untuk pembacaan data
